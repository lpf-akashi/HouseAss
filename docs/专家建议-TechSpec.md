总体判断：**这份规格书的大方向是对的**。以你的约束来看，`React + Vite + CloudBase + 高德地图 + 授权房产数据源` 是比较适合独立开发者的低运维方案。规格书也已经明确把腾讯云 CloudBase 作为核心 BaaS，包含静态托管、云函数、云数据库和认证能力，这与“零运维、国内可用、成本低”的目标基本一致。 但我建议你重点修改下面这些地方。

## 二、我建议保留的技术路线

**保留 CloudBase 作为主方案。** 腾讯云官方对 CloudBase 的定位就是云原生一体化开发环境，提供计算、存储、托管等 Serverless 能力，目标是减少服务器搭建和运维成本，这与你“零运维经验”的约束高度匹配。([腾讯云][3]) CloudBase 文档也明确包含数据库、云存储、云函数、静态网站托管、身份认证、HTTP 网关等能力，足够支撑豪斯助手的 MVP。([腾讯云开发 CloudBase][4])

我建议你的最终 MVP 技术栈这样写：

| 模块     | 推荐方案                            | 调整理由                                                             |
| -------- | ----------------------------------- | -------------------------------------------------------------------- |
| 前端     | React 18 + Vite                     | 保留，简单、资料多                                                   |
| UI       | Tailwind CSS 或 TDesign React       | Tailwind 灵活；如果你想更省事，可以用腾讯系 TDesign，组件更完整      |
| 部署     | CloudBase 静态网站托管              | 保留，国内访问和部署体验更适合你                                     |
| 后端     | CloudBase 云函数，Node.js 18/20     | 保留，但不要拆太细                                                   |
| 数据库   | CloudBase 文档数据库                | MVP 保留；后期复杂筛选再考虑 CloudBase PostgreSQL                    |
| 登录     | 匿名浏览 + 收藏/订阅时登录          | 比强制手机号登录更低成本、更低流失                                   |
| 地图     | 高德 Web 服务 API                   | 国内可用，但不要在文档里承诺固定免费额度；以控制台配额和授权政策为准 |
| 房产数据 | Mock/手工数据起步 + 授权 API 适配层 | 降低最大不确定性                                                     |
| AI 功能  | V0.1 不做，后期再接国内模型         | 避免合规、成本、备案复杂度                                           |

## 三、对你四个约束的逐项建议

### 1. 零运维经验

现在的 CloudBase 方案是合理的，但要进一步“傻瓜化”。文档中虽然说云函数一个模块一个函数即可，这个方向是对的，但 API 设计仍然偏传统后端。建议把 `/api/communities` 这种 REST 接口改成云函数内的 action 风格：

```js
callFunction("communities", {
  action: "search",
  keyword,
  city,
  filters
})
```

这样你不需要先理解 HTTP 网关、路由、中间件、CORS、自定义域名，学习成本更低。等产品跑通之后，再把高频接口迁到 HTTP 网关。

另外，文档里提到为了云函数冷启动可以“设置定时触发器保温”。我不建议初期做这个。冷启动多半只是几百毫秒级体验问题，但保温会增加调用和理解成本。MVP 阶段应该接受冷启动，用前端 loading 和缓存解决。

### 2. 国内环境可用

CloudBase、高德地图、微信订阅消息都是国内环境友好的选择。高德开放平台的 Web 服务覆盖搜索、定位、地理编码、路线规划、动态地图等能力，技术上足够支持“地铁距离、通勤时间、周边配套”这些功能。([高德开放平台][5])

但要注意：**不要把高德写成“5000 次/天免费”这种固定承诺**。高德当前页面已经把技术服务许可、基础配额、提额支持、超限支持等作为服务升级和授权的一部分展示，且相关技术服务许可规则从 2025 年 5 月 20 日开始执行。([高德开放平台][6]) 所以规格书里应该写：

> 高德地图 API 配额、QPS、商业授权和费用以高德开放平台控制台及最新服务协议为准。MVP 阶段通过服务端缓存、请求去重、按需计算降低调用量。

### 3. 成本控制

规格书里最大的问题是**免费额度写得太确定**。腾讯云官方当前 CloudBase 计费文档显示，免费体验环境是按资源点提供，每个云开发账号可创建一个免费体验版环境，每月提供 3000 点资源点；并且免费环境存在续费、功能限制和小程序发布后的到期规则。([腾讯云][2])

所以建议你把成本章节改成三层：

第一层，**MVP 零成本验证**：静态 Mock 数据 + CloudBase 免费体验环境 + 匿名浏览。

第二层，**小流量内测**：CloudBase 按量计费 + 预算告警 + 每日 API 调用上限。

第三层，**真实数据接入**：把房产 API 费用单独列为最大成本项，不和服务器成本混在一起。你原文已经意识到“数据源 API 费用取决于供应商”，但建议把它提升为“一号成本风险”。

### 4. 上手难度低

React + Vite 没问题，但建议你减少“高级工程化”。现在目录结构已经比较完整，但对初学者来说可以再简化：

```text
src/
  pages/
  components/
  services/cloudbase.js
  utils/

cloudfunctions/
  communities/
  users/
  dataSync/
```

不建议初期引入太多抽象，比如复杂 service 层、过多 hooks、过细云函数。你文档里“云函数不要拆太细”的建议是对的。

## 四、数据库设计需要补充的内容

你现在的集合设计覆盖了小区、成交、用户、收藏、订阅、通知、搜索历史，基本完整。 但建议补充以下字段和集合。

### 1. 给所有核心数据加来源和置信度

尤其是房产数据，必须能解释“从哪里来、何时更新、可信度如何”。

建议 `communities` 增加：

```js
sourceInfo: {
  provider: "manual | amap | beike | juhe | gov | other",
  sourceUrl: "",
  licenseType: "manual | authorized_api | public_data",
  fetchedAt: "",
  confidence: 0.85
}
```

### 2. 增加 `data_sync_jobs` 集合

用于可靠处理异步刷新。

```js
{
  _id: "auto",
  communityId: "wankeyuan",
  status: "pending | running | success | failed",
  lockedUntil: "",
  retryCount: 0,
  lastError: "",
  createdAt: "",
  updatedAt: ""
}
```

这样“按需刷新 + 防重复锁”才真正可靠。你现在已经有“同一小区 10 分钟内只允许触发一次更新”的想法，但还缺少可落地的数据结构。

### 3. 增加反馈与纠错集合

你文档里提到要增加“纠错反馈”入口，但数据库里还没有对应集合。建议加：

```js
feedbacks: {
  userId,
  communityId,
  type: "data_error | suggestion | complaint",
  content,
  status: "open | resolved | ignored",
  createdAt
}
```

### 4. 搜索历史要设置保留周期

`search_history` 会暴露用户关注区域、预算、购房意图，属于比较敏感的行为数据。建议默认只保留 30 或 90 天，并提供“清空历史”功能。

## 五、合规和产品表达需要补充

你这个项目涉及房价、成交、小区评价、购房决策，不能只写技术规格。建议补充一个“合规与免责声明”章节。

必须补充：

1. 数据来源声明：每个小区详情页展示“数据来源、更新时间、仅供参考”。
2. 不构成购房/投资建议：把“可去看/谨慎/暂不建议”改成“看房优先级/关注风险”。
3. 隐私政策：说明手机号、收藏、订阅、搜索历史如何收集、保存和删除。
4. 数据删除入口：用户可以删除收藏、订阅、通知和搜索历史。
5. 第三方 API 合规：所有房产数据必须来自授权 API、公开数据或人工录入，不直接爬取竞品平台。你文档里已经把“抓取房产平台数据存在法律风险”列为高风险，这是正确的。

如果后续加入 AI 问答或“AI 帮你判断小区”，还要额外考虑生成式 AI 服务合规。国家网信办《生成式人工智能服务管理暂行办法》适用于向境内公众提供生成文本、图片、音频、视频等内容的服务，并要求保护用户输入信息和使用记录，不得收集非必要个人信息。([国家市场监督管理总局][7]) 所以 V0.1 阶段我建议先不要做大模型问答。

## 六、我给你的最终建议

你的技术规格书可以保留主架构，但需要改成下面这个定位：

> V0.1 不追求“真实全量房产数据平台”，而是做一个“低成本、低运维、半静态数据驱动的看房决策助手”。先用 Mock/手工数据验证产品体验，后续再通过授权 API 接入真实数据。

最推荐的落地顺序是：

1. **第一阶段**：React + CloudBase 静态托管 + Mock 小区数据，完全零成本上线。
2. **第二阶段**：接入 CloudBase 文档数据库，支持搜索、详情、收藏、订阅。
3. **第三阶段**：接入高德地图，只做通勤、地铁距离、周边配套，并做服务端缓存。
4. **第四阶段**：再谈真实房产数据 API，先确认价格、授权、字段、调用限制。
5. **第五阶段**：用户量起来后，再考虑小程序、AI 解读、复杂推荐算法。

一句话总结：**CloudBase 方向对，但现在的规格书对“免费额度、真实房产数据、异步刷新、手机号登录、REST API”估计得偏乐观。你应该把项目收缩成“Web 优先、匿名优先、Mock 数据优先、授权数据源后置”的方案，这样最符合你的四个约束。**
